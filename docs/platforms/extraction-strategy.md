# Стратегия извлечения данных со страницы

CSS-селекторы (`#searchInput`, `.es-el-name`) ломаются при каждом редизайне. Вместо них — **трёхуровневая** схема.

## Уровни (от надёжного к запасному)

| Уровень | Что используем | Когда |
|---------|----------------|-------|
| 1. **Leaf-поля** | Атрибуты `content="leaf:…"` на узлах DOM | Сбербанк-АСТ биндит данные в разметку; стабильнее классов |
| 2. **Текстовые блоки** | `main.innerText`, метки «Наименование…:» + значение на следующей строке | Карточка тендера, fallback для выдачи |
| 3. **Семантика UI** | `get_by_role`, `get_by_placeholder`, текст «Найдено процедур» | Клики и ожидания, не парсинг |

Опционально позже:

| 4. **LLM по снимку** | Сжатый `PageSnapshot.main_text` + JSON schema | Если leaf и текст не дали полей; включается флагом |

## Модули

- `browser/page_context.py` — снимок страницы, leaf-листинг, семантические действия
- `browser/text_blocks.py` — парсинг меток и разбиение выдачи по `№`
- Адаптер площадки — только оркестрация шагов, без CSS в коде

## PageSnapshot

```python
snapshot = await capture_snapshot(page)
# snapshot.main_text      — текст основной области
# snapshot.listing_items  — из leaf:objectHrefTerm, purchName, …
# snapshot.results_marker — «Найдено процедур 16 …»
```

При ошибках можно сохранять `main_text` в `data/debug/` для ручного разбора.

## Принципы

1. **Действия** — по смыслу (роль, placeholder, видимый текст), не по id/class.
2. **Данные** — сначала встроенная схема сайта (`leaf:*`), потом текст по меткам.
3. **Агент/LLM** — только fallback, не в hot path каждого лота (стоимость и задержка).
